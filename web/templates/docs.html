Como usar Inspector

Inspector te permite crear endpoints de prueba, inspeccionar peticiones entrantes en tiempo real y enviar peticiones HTTP o WebSocket para pruebas de integracion.

Inicio rapido

  1. Inicia sesion con tus credenciales.
  2. Ve a Endpoints y crea un endpoint con Name y Slug.
  3. Copia la URL generada y enviale peticiones desde tu sistema.
  4. Abre Requests para ver cada solicitud capturada.
  5. En Dashboard revisa actividad reciente y estadisticas.

Herramienta Sender

  • Usa Sender para probar APIs HTTP sin salir de Inspector.
  • Puedes enviar headers en JSON o formato Key: Value.
  • El historial queda guardado en History para reintentos.
  • WS Client sirve para conectarte y probar servicios WebSocket.

Ejemplos utiles

Enviar un webhook con curl:

curl -X POST "http://TU_HOST:PUERTO/in/mi-endpoint" \
  -H "Content-Type: application/json" \
  -d '{"event":"order.created","id":123}'

Conectar a un endpoint WebSocket:

ws://TU_HOST:PUERTO/in/mi-endpoint/ws

Buenas practicas

  • Usa slugs descriptivos por entorno, por ejemplo pagos-staging.
  • Limpia logs desde Endpoints si quieres sesiones de prueba mas limpias.
  • No compartas credenciales de login en ambientes publicos.
  • Si usas HTTPS con certificados de desarrollo, verifica la configuracion TLS del destino.

Mock Rules — respuestas condicionales por endpoint

Las Mock Rules te permiten que un mismo endpoint responda de forma distinta dependiendo del request recibido: metodo HTTP, path, query params, headers o body. Es similar a Beeceptor o Mockoon. Cada endpoint puede tener varias reglas; gana la primera que coincide (por prioridad ascendente). Si ninguna coincide, se usa la respuesta estatica del endpoint como fallback.

Como crear una regla

  1. Ve a Endpoints y abre cualquier endpoint.
  2. Verás el panel Mock Rules al final de la tarjeta.
  3. Completa el formulario "Nueva Mock Rule" y haz clic en Create Mock Rule.
  4. La regla aparece en la lista inmediatamente y está activa por defecto.
  5. Puedes editarla inline y guardar con Save Rule.

Campos clave

  • Priority — número menor = se evalúa primero. Usa 1, 10, 100…
  • Method — GET, POST, ANY (o vacío para cualquiera).
  • Path mode — any · exact · prefix · regex.
  • Body mode — any · exact · contains · regex · json (subconjunto de campos).
  • Delay ms — retraso antes de responder (útil para simular latencia).
  • Active — toggle rápido sin borrar la regla.

Ejemplo: 200 para GET, 404 para el resto

Regla 1 (priority 1):

Method: GET
Path mode: any
Response: 200 {"ok":true}

Fallback del endpoint:

Response: 404 {"error":"not found"}

Ejemplo: respuesta según body JSON

Method: POST
Body mode: json
Body pattern:
  {"type":"payment"}
Response: 201
  {"charged":true}

Ejemplo: simular timeout

Method: ANY
Path mode: any
Delay ms: 5000
Response: 503
  {"error":"timeout"}
El orden de evaluación es Priority ASC. Si dos reglas tienen la misma prioridad, gana la de menor ID (la más antigua). El fallback final es siempre la respuesta configurada en el endpoint.
