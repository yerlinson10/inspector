How to use Inspector

Inspector lets you create test endpoints, inspect incoming requests in real time, and send HTTP or WebSocket requests for integration testing.

Quick Start

  1. Sign in with your credentials.
  2. Go to Endpoints and create an endpoint with Name and Slug.
  3. Copy the generated URL and send requests from your system.
  4. Open Requests to inspect each captured request.
  5. Use Dashboard to review recent activity and statistics.

Sender Tool

  • Use Sender to test HTTP APIs without leaving Inspector.
  • Headers and body are validated as JSON before sending.
  • Each JSON field includes actions: Validate, Fix, and Example.
  • History stores sent requests for retries.
  • WS Client helps you connect to and test WebSocket services.

Useful Examples

Send a webhook with curl:

curl -X POST "http://TU_HOST:PUERTO/in/mi-endpoint" \
  -H "Content-Type: application/json" \
  -d '{"event":"order.created","id":123}'

Connect to a WebSocket endpoint:

ws://TU_HOST:PUERTO/in/mi-endpoint/ws

Best Practices

  • Use descriptive slugs per environment, for example payments-staging.
  • Clear logs from Endpoints when you need cleaner test sessions.
  • Do not share login credentials in public environments.
  • If you use HTTPS with development certificates, verify TLS configuration on the target service.

Mock Rules - conditional responses per endpoint

Mock Rules let the same endpoint return different responses depending on the incoming request: HTTP method, path, query params, headers, or body. Similar to Beeceptor or Mockoon, each endpoint can have multiple rules; the first matching rule wins (ascending priority). If none matches, the endpoint static response is used as fallback.

How to create a rule

  1. Go to Endpoints and open any endpoint.
  2. You will see the Mock Rules panel at the bottom of the card.
  3. Fill in the "New Mock Rule" form and click Create Mock Rule.
  4. The rule appears immediately in the list and is active by default.
  5. You can edit it inline and save with Save Rule.

Key fields

  • Priority - lower number = evaluated first. Use 1, 10, 100...
  • Method - GET, POST, ANY (or empty for any).
  • Path mode — any · exact · prefix · regex.
  • Body mode - any · exact · contains · regex · json (field subset matching).
  • Delay ms - delay before responding (useful to simulate latency).
  • Active - quick toggle without deleting the rule.

Example: 200 for GET, 404 for everything else

Rule 1 (priority 1):

Method: GET
Path mode: any
Response: 200 {"ok":true}

Endpoint fallback:

Response: 404 {"error":"not found"}

Example: response based on JSON body

Method: POST
Body mode: json
Body pattern:
  {"type":"payment"}
Response: 201
  {"charged":true}

Example: simulate timeout

Method: ANY
Path mode: any
Delay ms: 5000
Response: 503
  {"error":"timeout"}
The evaluation order is Priority ASC. If two rules share the same priority, the lower ID (older rule) wins. Final fallback is always the endpoint configured response.
